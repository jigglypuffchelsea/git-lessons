Git 手把手教學

目標：搞懂 git 是什麼，會用基本指令記錄程式碼的每一個版本，改壞了也能回去。
時間：順利的話大概 15 分鐘
範圍：git init、git status、git diff、git add、git commit、git log，還有 Claude Code 的 /commit。

Git 到底是什麼？

你程式碼的存檔系統。RPG 打 Boss 之前會先存檔吧？Git 就是這樣——覺得「這個版本還不錯」就存一個點，之後任何時候都可以回去。

全部都在你自己的電腦上發生，不需要網路，不需要帳號。

你需要準備的東西

  • Git（版本控制工具本人）：確認有沒有 → 打開 Terminal，輸入 git --version。看到 git version 2.x.x 就有了。沒有 → 輸入 xcode-select --install 跟著步驟走。
  • Terminal（打指令的視窗）：按 Cmd + Space，搜尋「Terminal」。
  • 一個專案資料夾：裡面有你正在寫的程式碼。還沒有的話，隨便建一個資料夾放幾個檔案也行。

先搞定 Terminal 基礎，60 秒

你不需要很懂 Terminal，知道怎麼去正確的地方就夠了。

1 打開 Terminal

做這個：開 Terminal。
在哪裡：按 Cmd + Space，打 "Terminal"，按 Enter。
你會看到：一個視窗，裡面有一行文字，最後有閃爍的游標。
確認：視窗打開，游標在閃。
沒出現的話：去 Applications → Utilities → Terminal。

2 去你的專案資料夾

做這個：輸入 cd 加上你的專案路徑。
在哪裡：Terminal 視窗裡。
輸入這個：
cd /Users/你的名字/Projects/你的專案名稱
你會看到：游標跑到下一行，沒有任何錯誤。
確認：輸入 pwd 會顯示你現在在哪裡，應該是你的專案路徑。
沒出現的話：在 Finder 找到你的專案資料夾，在 Terminal 打 cd （空格），然後把資料夾拖進 Terminal 視窗裡，按 Enter。

開始

3 建立 git 倉庫

做這個：輸入 git init。
在哪裡：Terminal 視窗裡，確認你在專案資料夾內。
輸入這個：
git init
你會看到：
Initialized empty Git repository in /Users/你的名字/Projects/你的專案名稱/.git/
確認：有看到 Initialized 這個字。
沒出現的話：確認你在正確的資料夾裡（回去第 2 步）。
這一步只需要做一次。它會在你的資料夾裡建一個隱藏的 .git 資料夾，git 所有的紀錄都存在裡面。之後不用再跑第二次。
如果你的專案已經有 .git 資料夾（例如從別人那邊拿來的），跳過這步。怎麼確認？輸入 git status，沒有噴 fatal: not a git repository 就是已經有了。

4 看看現在的狀態

做這個：輸入 git status。
在哪裡：Terminal 視窗裡。
輸入這個：
git status
你會看到：
On branch main
Untracked files:
        src/pages/booking.tsx
        src/utils/helpers.ts

或是 nothing to commit, working tree clean（表示你最近沒改東西）。

確認：沒有看到 fatal: 開頭的錯誤。
沒出現的話：如果看到 fatal: not a git repository，回去第 3 步跑 git init。

看懂那些字在說什麼：

  • Untracked files → 新檔案，git 還沒開始追蹤它
  • modified: xxx → 這個檔案你改過了，但還沒存到 git 歷史紀錄
  • nothing to commit → 沒有任何改動

5 看看你到底改了什麼

做這個：輸入 git diff。
在哪裡：Terminal 視窗裡。
輸入這個：
git diff
你會看到：
- const name = "舊的內容";
+ const name = "新的內容";

前面有 - 的是刪掉的行（紅色），+ 的是新加的行（綠色）。

確認：有看到你改過的內容。如果什麼都沒顯示，表示沒有已追蹤檔案被修改（新檔案不會出現在 git diff）。
沒出現的話：畫面被佔滿了的話，按 q 離開。
這步不是必要的，但養成 commit 之前先看一眼 diff 的習慣很好。確認你存的是你真的想存的東西。

6 把改動放進暫存區

做這個：輸入 git add .（注意最後有個點）。
在哪裡：Terminal 視窗裡。
輸入這個：
git add .
你會看到：沒有任何輸出——這是正常的，代表成功。
確認：再輸入一次 git status，剛才的檔案會出現在 Changes to be committed: 底下。
沒出現的話：確認有打那個點（.），點代表「全部改過的檔案」。
暫存區（Staging Area）是什麼？想像你要寄包裹：先把東西擺到桌上（git add），確認都放好了，才打包寄出（git commit）。那個桌子就是暫存區。

7 存一個紀錄點

做這個：輸入 git commit -m 加上你的訊息。
在哪裡：Terminal 視窗裡。
輸入這個：
git commit -m "新增預約頁面的表單驗證"

（引號裡換成你這次改了什麼）

你會看到：
[main 3a9f2c1] 新增預約頁面的表單驗證
 1 file changed, 24 insertions(+), 3 deletions(-)
確認：有看到 [main xxxxxxx] 開頭的那行。
沒出現的話：確認 -m 後面有引號，訊息在引號裡面。

8 看看歷史紀錄

做這個：輸入 git log --oneline。
在哪裡：Terminal 視窗裡。
輸入這個：
git log --oneline
你會看到：
3a9f2c1 新增預約頁面的表單驗證
abc1234 修正登入頁面的 bug
def5678 初始化專案
確認：你剛才存的那條紀錄出現在最上面。
沒出現的話：如果畫面被佔滿進入翻頁模式，按 q 離開。

9 用 Claude Code 幫你 commit

做這個：在 Claude Code 的對話框裡輸入 /commit。
在哪裡：Claude Code 的輸入框。
輸入這個：
/commit
你會看到：Claude 分析你改了什麼，自動寫好 commit 訊息，問你要不要確認。
確認：訊息看起來合理，按確認送出。
沒出現的話：確認你有先存好所有開啟的檔案（Cmd + S）。
這步等於幫你做了第 6 步 + 第 7 步，而且 commit 訊息它幫你想好。懶得自己寫訊息的時候用這招。

搞定了。你剛剛做了什麼？

Git 幫你記錄程式碼的每一個版本，全部存在你自己的電腦上。git init 是開啟這個功能，git add 是選要記哪些改動，git commit 是存到歷史紀錄。每個 commit 就是一個存檔點，改壞了可以回去。

可能會出包的地方

fatal: not a git repository
你不在一個有 git 的資料夾裡。先 cd 到你的專案資料夾，然後跑 git init。
nothing to commit, working tree clean
還沒改任何東西，或改了但還沒存檔。先按 Cmd + S 存一下。
error: src refspec main does not match any
還沒做過任何 commit。先跑 git add . → git commit -m "第一次 commit"。
commit 的訊息要寫什麼？
寫你「改了什麼」就好。「修正登入 bug」「新增預約頁面」「更新首頁文案」——讓未來的自己看到知道當時在幹嘛就行了。或是用 /commit 讓 Claude 幫你寫。
按了 git diff 或 git log 之後畫面卡住？
你進入了分頁模式。按 q 就能離開，回到正常的 Terminal。

第一章：安全網——改壞了怎麼辦

學 git 最怕的就是「我是不是把東西搞壞了回不去」。這章教你各種「後悔藥」，學完之後你可以放心大膽地改東西。

1 告訴 git 哪些東西不用管

做這個：在專案根目錄建一個叫 .gitignore 的檔案。
在哪裡：你的專案資料夾最頂層。
輸入這個：
# 在 Terminal 輸入（或用編輯器建立檔案也行）
touch .gitignore

然後打開這個檔案，寫入你要忽略的東西：

# 套件依賴（超大，不應該進 git）
node_modules/

# 環境變數（有密碼，絕對不能進 git）
.env
.env.local

# macOS 系統檔案（垃圾）
.DS_Store

# 編輯器設定
.vscode/
.idea/
你會看到：之後跑 git status，被忽略的檔案不會再出現在列表裡。
確認：git status 裡面看不到 node_modules 和 .env。
沒出現的話：確認檔名是 .gitignore（前面有點），而且放在專案最頂層。
為什麼要忽略？兩個原因：（1）node_modules 動輒幾萬個檔案，推到 GitHub 會爆掉；（2）.env 有密碼和 API key，推上去等於把鑰匙公開給全世界。
已經被 git 追蹤的檔案，加進 .gitignore 不會自動消失。要先跑 git rm --cached 檔名 取消追蹤，再 commit。

2 還沒 commit 的改動，想丟掉

做這個：用 git restore 把檔案還原到上一次 commit 的狀態。
在哪裡：Terminal 視窗裡。
輸入這個（還原單一檔案）：
git restore src/pages/booking.tsx
輸入這個（還原全部改動）：
git restore .
你會看到：沒有任何輸出——代表成功。去看那個檔案，內容回到改之前的樣子。
確認：git status 裡面那個檔案消失了（不再顯示 modified）。
沒出現的話：確認檔案路徑打對了。用 git status 看正確的檔名。
這個操作沒有回頭路——還原之後，你改過的東西就真的不見了。所以用之前先確認你真的不要那些改動。如果不確定，用下一步的 git stash 會更安全。

已經 git add 了怎麼辦？

  • 如果你已經把改動放進暫存區（跑了 git add），要先把它退出來：
# 把檔案從暫存區退回去（改動還在，只是退出暫存）
git restore --staged src/pages/booking.tsx

# 然後再丟掉改動
git restore src/pages/booking.tsx

3 改到一半，先放著等下回來

做這個：用 git stash 把目前的改動「暫時收起來」。
在哪裡：Terminal 視窗裡。
輸入這個：
git stash
你會看到：
Saved working directory and index state WIP on main: 3a9f2c1 你的 commit 訊息
✎
改到一半
⟶
📦
git stash
收進口袋
⟶
✓
乾淨工作區
去做別的事
⟶
✎
git stash pop
拿回來繼續
確認：git stash 之後跑 git status 會看到 nothing to commit。git stash pop 之後改動又回來了。
沒出現的話：如果看到 No local changes to save，表示你沒有任何改動可以收。
想像你在煮飯，煮到一半有人按門鈴。你把鍋子從爐子上移開（git stash），去開門處理完（做別的事），回來再把鍋子放回爐子上（git stash pop）。

4 已經 commit 了，但想反悔

做這個：用 git reset 把最近的 commit 撤回。
在哪裡：Terminal 視窗裡。
輸入這個（保留改動，只是撤回 commit）：
git reset --soft HEAD~1
你會看到：沒有輸出。但跑 git log --oneline 會發現最後一筆 commit 消失了，而那些改動回到了暫存區。
確認：git log --oneline 最上面不再是剛才那筆 commit。git status 顯示那些檔案在 Changes to be committed。
沒出現的話：確認 HEAD~1 的格式——波浪號加數字 1，表示「往前一步」。
HEAD~1 是「最近一筆 commit」的意思。HEAD~2 就是「最近兩筆」。--soft 表示「撤回 commit 但保留改動」。還有一個 --mixed（預設值），差別是改動會退回到「未暫存」的狀態。
這裡故意不教 --hard，因為它會直接刪掉你的改動，沒有回頭路。等你更熟練之後再學。

這章的新術語：

  • .gitignore → 告訴 git「不要管這些檔案」的清單
  • git restore → 把還沒 commit 的改動丟掉，還原檔案
  • git stash → 暫時收起改動，像按了暫停鍵
  • git reset → 撤回已經做的 commit
  • HEAD → 目前最新的那一筆 commit，你的「現在位置」
  • --staged → 指「已經 git add 放進暫存區的」

第二章：推到 GitHub

到目前為止，你的所有 commit 都存在自己電腦上。這章教你把它推到 GitHub——雲端備份、跟別人協作、或是讓 Claude Code 幫你部署都需要這一步。

5 在 GitHub 上建一個 Repository

做這個：去 GitHub 網站建立一個新的 repository。
在哪裡：瀏覽器，打開 github.com。
步驟：
  1. 登入 GitHub（沒有帳號就先註冊一個，免費的）
  2. 右上角按 + → New repository
  3. Repository name：輸入你的專案名稱（例如 my-clinic-app）
  4. 選 Private（私人的，別人看不到）或 Public（公開的）
  5. 其他選項都不要勾，直接按 Create repository
你會看到：一個頁面，上面有一堆指令。我們需要的是「push an existing repository」那段。
確認：頁面上有顯示一個網址，長得像 https://github.com/你的帳號/你的專案名稱.git。
Repository 就是你的專案在 GitHub 上的家。你本機有一份，GitHub 上也有一份，兩邊透過 push 和 pull 同步。

6 把本機跟 GitHub 連起來

做這個：用 git remote add 告訴本機的 git「你的 GitHub 位址在這裡」。
在哪裡：Terminal 視窗裡，確認你在專案資料夾內。
輸入這個：
git remote add origin https://github.com/你的帳號/你的專案名稱.git

（把網址換成第 5 步你看到的那個）

你會看到：沒有任何輸出——代表成功。
確認：輸入 git remote -v，會看到 origin 和你的 GitHub 網址。
沒出現的話：如果說 remote origin already exists，表示已經連過了。用 git remote -v 確認網址對不對。
origin 是一個名字，代表「遠端那台伺服器」。你可以取別的名字，但大家習慣叫 origin，就像大家習慣把主要分支叫 main 一樣。

7 把 commit 推到 GitHub

做這個：用 git push 把你本機的 commit 傳到 GitHub。
在哪裡：Terminal 視窗裡。
輸入這個（第一次推要加 -u）：
git push -u origin main
你會看到：
Enumerating objects: 15, done.
Counting objects: 100% (15/15), done.
Writing objects: 100% (15/15), 3.21 KiB | 3.21 MiB/s, done.
 * [new branch]      main -> main
Branch 'main' set up to track remote branch 'main' from 'origin'.
確認：去 GitHub 重新整理頁面，你的程式碼出現了。
沒出現的話：如果要求輸入帳號密碼，GitHub 現在不接受密碼了。你需要設定 Personal Access Token 或 SSH Key。最快的方法是安裝 gh（GitHub CLI），然後跑 gh auth login 跟著步驟走。
-u 是「記住這個連結」的意思。第一次推的時候加上去，之後就只需要打 git push 就好，不用再重複寫 origin main。

8 把別人的改動拉下來

做這個：用 git pull 把 GitHub 上的最新版本拉到你的電腦。
在哪裡：Terminal 視窗裡。
輸入這個：
git pull
你會看到（如果有新東西）：
remote: Enumerating objects: 5, done.
Updating 3a9f2c1..b7c8d3e
Fast-forward
 src/components/Header.tsx | 12 ++++++------
 1 file changed, 6 insertions(+), 6 deletions(-)

或是 Already up to date.（表示你已經是最新的）。

確認：沒有看到錯誤訊息。
沒出現的話：如果說 There is no tracking information，跑 git pull origin main。
養成習慣：每次開始工作之前先 git pull。這樣你手上永遠是最新版本，減少之後衝突的機會。

9 從 GitHub 抓一個現有專案

做這個：用 git clone 把 GitHub 上的專案整個下載到你的電腦。
在哪裡：Terminal 視窗裡，先 cd 到你想放專案的地方（例如 cd ~/Projects）。
輸入這個：
git clone https://github.com/某人/某專案.git
你會看到：
Cloning into '某專案'...
remote: Enumerating objects: 156, done.
Resolving deltas: 100% (87/87), done.
確認：ls 看一下，多了一個跟專案同名的資料夾。cd 進去之後，git log --oneline 能看到歷史紀錄。
git clone 等於幫你做了 git init + git remote add + 把所有東西下載好。拿別人的專案或是換電腦的時候用這招。

這章的新術語：

  • Repository (repo) → 專案的倉庫，包含所有檔案和歷史紀錄
  • Remote → 遠端伺服器，通常就是 GitHub
  • origin → 遠端伺服器的預設名稱
  • git push → 把本機的 commit 傳到 GitHub
  • git pull → 把 GitHub 的最新版拉到本機
  • git clone → 把 GitHub 上的整個專案下載到本機

第三章：Branch——平行宇宙

到目前為止你所有的 commit 都在 main 這條線上。但如果你想做一個新功能，又不想影響目前正常運作的版本呢？開一個 branch。

想像你在寫小說，寫到第十章覺得「如果主角在這裡做了不同的選擇呢？」你不會把原稿擦掉重寫——你會拿一張新的紙，從第十章開始寫另一個版本。Branch 就是那張新的紙。

Branch 運作方式
main
A
B
C
feature
D
E
feature 從 C 分出去，D 和 E 只存在 feature 分支，不影響 main

10 建立一個新分支

做這個：用 git switch -c 建立一個新分支並切換過去。
在哪裡：Terminal 視窗裡。
輸入這個：
git switch -c feature/add-booking-form

（feature/add-booking-form 是分支名稱，換成你要做的事）

你會看到：
Switched to a new branch 'feature/add-booking-form'
確認：git branch 會列出所有分支，你的新分支前面有 * 號，代表你現在在那裡。
沒出現的話：確認你至少有一筆 commit。全新的空 repo 沒辦法開分支。
分支命名慣例：feature/功能名稱、fix/修什麼、hotfix/緊急修復。不是硬規定，但這樣一看名字就知道這個分支在幹嘛。

11 在分支之間切換

做這個：用 git switch 切換到不同的分支。
在哪裡：Terminal 視窗裡。
輸入這個：
# 切回 main
git switch main

# 切到你的功能分支
git switch feature/add-booking-form
你會看到：
Switched to branch 'main'
確認：切換之後，打開你的程式碼檔案——它會變成那個分支的版本。
沒出現的話：如果說 error: Your local changes would be overwritten，表示你有未 commit 的改動。先 git stash 收起來（第 3 步），或是 git add + git commit 存起來，再切換。
切換分支的時候，你的檔案內容會跟著變。不要慌，你在 A 分支做的事不會影響 B 分支——它們是獨立的平行宇宙。

12 在分支上做事然後 commit

做這個：確認你在功能分支上，改東西，然後 commit。
在哪裡：Terminal 視窗裡。
輸入這個：
# 確認你在哪個分支
git branch

# 改完程式碼之後，照之前學的流程
git add .
git commit -m "新增預約表單的日期選擇器"

# 要推到 GitHub 的話（第一次要加 -u）
git push -u origin feature/add-booking-form
你會看到：跟之前一樣的 commit 成功訊息。
確認：git log --oneline 看到你的新 commit。切回 git switch main 之後，這個 commit 不在 main 上——它只存在你的功能分支。
這就是 branch 的價值：你在功能分支上怎麼改都不會影響 main。等做完、測好了，再合併回去。

這章的新術語：

  • Branch（分支） → 一條獨立的開發線，跟主線互不影響
  • main → 預設的主分支，通常代表「穩定版本」
  • git switch -c → 建立新分支並切換過去（-c = create）
  • git switch → 切換到已存在的分支
  • git branch → 列出所有分支，* 標記你目前在哪

第四章：Merge——合併你的平行宇宙

功能做完了，測試也通過了。現在要把你的功能分支合併回 main，讓正式版本也有這個新功能。

13 把分支合併回 main

做這個：先切回 main，然後用 git merge 把功能分支合併過來。
在哪裡：Terminal 視窗裡。
輸入這個：
# 第一步：切回 main
git switch main

# 第二步：把功能分支合併進來
git merge feature/add-booking-form
你會看到（順利的話）：
Updating 3a9f2c1..e5f6g7h
Fast-forward
 src/components/BookingForm.tsx | 45 +++++++++++++++++++++
 1 file changed, 45 insertions(+)
確認：git log --oneline 在 main 上看到功能分支的 commit 了。
沒出現的話：確認你已經切回 main（git branch 確認 * 在 main）。
合併的方向很重要：你要站在「接收方」去合併「來源方」。所以是先切到 main，再 merge 功能分支——不是反過來。

14 兩種合併方式（看懂就好）

這步不用輸入指令，純粹讓你看懂 git 在做什麼。

Fast-forward（快轉）：

main 沒有新的 commit，功能分支直接往前走。Git 只要把 main 的指標往前移就好，像是快轉錄音帶。

Fast-forward 合併
main
A
B
C
D
E
main 直接快轉到 E，一條直線

三方合併（Three-way merge）：

main 和功能分支都有新的 commit，git 需要建立一個新的「合併 commit」把兩邊的改動整合在一起。

三方合併
main
A
B
C
F
G
feature
D
E
G 是合併 commit，整合了 main 的 F 和 feature 的 D、E
你不用選——git 會自動判斷用哪一種。大部分時候你不需要管這些細節，只要知道「兩邊都有改動的時候，git 會幫你合」就好。

15 衝突解決：兩個人改了同一行

什麼時候會發生：你在功能分支改了第 10 行，同時 main 上也有人改了第 10 行。git 不知道該用哪個版本，就會停下來叫你自己決定。
你會看到：
Auto-merging src/components/Header.tsx
CONFLICT (content): Merge conflict in src/components/Header.tsx
Automatic merge failed; fix conflicts and then commit the result.

打開那個檔案，你會看到 git 標記的衝突區域：

main 的版本 feature 的版本 <<<<<<< HEAD const title = "預約管理系統"; ======= const title = "智慧預約平台"; >>>>>>> feature/add-booking-form

看懂衝突標記：

  • <<<<<<< HEAD → 你目前所在的分支（main）的版本
  • ======= → 分隔線
  • >>>>>>> feature/... → 要合併進來的分支的版本
做這個：
  1. 打開有衝突的檔案
  2. 決定你要留哪個版本（或兩個都留、或寫一個全新的）
  3. 刪掉 <<<<<<<、=======、>>>>>>> 這三行標記
  4. 存檔

例如，你決定用新的名稱，改完之後檔案長這樣：

const title = "智慧預約平台";
然後：
# 告訴 git「我處理好了」
git add src/components/Header.tsx

# 完成合併
git commit -m "合併 feature/add-booking-form，採用新標題"
確認：git status 不再顯示 Unmerged paths。git log --oneline 最上面是你的合併 commit。
卡住的話：如果衝突太複雜想放棄這次合併，跑 git merge --abort 回到合併前的狀態。
衝突不可怕，就是 git 在問你：「兩邊都改了，你說了算。」看清楚兩邊的改動，選你要的版本，刪掉標記，存檔，commit。就這樣。

這章的新術語：

  • git merge → 把一個分支的改動合併到目前的分支
  • Fast-forward → 最簡單的合併，直接快轉
  • Merge conflict（衝突） → 兩邊改同一行，git 需要你手動決定
  • git merge --abort → 放棄合併，回到合併前的狀態

Cheat Sheet

你現在會了：改壞了知道怎麼回去、程式碼推到 GitHub 備份、開 branch 做新功能不影響主線、做完合併回去、遇到衝突也知道怎麼處理。這些就是日常開發 90% 的 git 操作。

可能會出包的地方

remote: Permission denied / 403
GitHub 認證有問題。最快解法：安裝 GitHub CLI，跑 gh auth login，跟著步驟走。
error: failed to push some refs
GitHub 上有你本機沒有的新 commit。先跑 git pull 把最新版拉下來，再 git push。
Your local changes would be overwritten by checkout
你有未 commit 的改動，git 不讓你切分支。先 git stash 或 git add + git commit。
CONFLICT (content): Merge conflict in...
不要慌。打開檔案，找到 <<<<<<< 標記，決定要留哪邊，刪掉標記，存檔，git add + git commit。
分支用完了要刪嗎？
合併完之後可以刪。git branch -d feature/add-booking-form。用小寫 -d，git 會確認已經合併才讓你刪。
什麼時候該開新分支？
做新功能、修 bug、任何「不確定會不會搞壞東西」的改動。寧可多開分支，也不要在 main 上賭運氣。
push 跟 commit 有什麼不同？
commit 是存到你自己電腦的歷史紀錄。push 是把那些紀錄傳到 GitHub。你可以 commit 十次之後再一次 push。

下一步可以學的東西

  • Pull Request (PR)：在 GitHub 上發起合併請求，讓隊友先 review 你的程式碼再合併。團隊協作的標準流程。
  • git rebase：另一種合併方式，讓歷史紀錄更乾淨。進階技巧，先會 merge 再學 rebase。
  • GitHub Actions / CI/CD：每次 push 自動跑測試、自動部署。讓機器人幫你做重複的事。
  • git cherry-pick：從別的分支挑選特定的 commit 合併過來，不用整個分支都合。
